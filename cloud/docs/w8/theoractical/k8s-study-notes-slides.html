Core Architecture & Primitives.

Giải mã cơ chế vận hành cốt lõi, kiến trúc cụm máy chủ và cách thức hoạt động của các đối tượng cơ bản.

Block 01 · Bài Toán Vận Hành

Tại sao chúng ta cần Kubernetes?

Docker giải quyết môi trường chạy local, nhưng quy mô sản xuất (production) phát sinh nhiều rủi ro cần giải pháp tự động hóa.

Vấn đề thực tế Khi không có Kubernetes Giải pháp với Kubernetes Hạn chế (Drawbacks)
Container sập lúc nửa đêm Kỹ sư phải SSH vào server để restart thủ công. Tự phát hiện và restart tự động qua Control Loop. Gây vòng lặp CrashLoopBackOff nếu lỗi logic app.
Traffic đột biến (spike 5x) Scale hạ tầng bằng tay, chậm trễ dẫn tới sập mạng. HPA tự động scale số lượng Pod dựa trên CPU/RAM thực tế. Độ trễ khởi động pod mới khiến app vẫn chịu tải trong vài phút.
Cập nhật phiên bản mới Gây downtime hoặc cấu hình kịch bản đổi cổng phức tạp. Rolling update thay thế dần từng Pod giúp zero-downtime. Yêu cầu ứng dụng phải tương thích ngược (backward compatible).
Phân bổ container lên cụm Tính toán tài nguyên trống thủ công, dễ mất cân bằng. Scheduler tự động lập lịch tối ưu dựa trên cấu hình khai báo. Cụm quá tải khiến pod mới bị kẹt ở trạng thái Pending.

Declarative State & Vòng Lặp Control Loop.

Sự khác biệt giữa việc ra lệnh từng bước và việc khai báo trạng thái mong muốn.

Imperative (Mệnh lệnh) vs Declarative (Khai báo)
Imperative: Ra lệnh chạy trực tiếp container. Nếu sập, lập trình viên tự quản trị luồng khôi phục.
# Chạy thủ công từng container docker run -d nginx:1.27
Declarative: Khai báo trạng thái mong muốn (Desired State) qua YAML. Kubernetes tự điều khiển.
# Khai báo cấu hình YAML kubectl apply -f deployment.yaml
Vòng Lặp Reconcile (Control Loop)
Mỗi controller trong K8s chạy vòng lặp này liên tục để đồng bộ trạng thái thực tế về khớp với YAML khai báo.

Block 01 · Thành Phần Hệ Thống

Khám Phá Kiến Trúc Cluster.

Sự phân chia rõ ràng giữa bộ não điều phối (Control Plane) và cơ bắp thực thi (Worker Nodes).

Control Plane (Bộ não điều phối)
kube-apiserver
Cổng giao tiếp duy nhất, xác thực quyền truy cập RBAC và ghi cấu hình tài nguyên vào etcd.
etcd
Cơ sở dữ liệu key-value phân tán sử dụng thuật toán Raft, lưu trữ toàn bộ trạng thái của cụm.
kube-scheduler
Quan sát Pod mới chưa gán Node, chọn Node tối ưu dựa trên cấu hình tài nguyên yêu cầu (requests).
kube-controller-mgr
Nơi tập hợp các bộ điều phối (Deployment Controller, Node Controller) duy trì reconcile loop.
Worker Node (Cơ bắp thực thi)
kubelet
Agent chạy trực tiếp trên mỗi Node, giám sát và ra lệnh cho Container Runtime khởi tạo container.
kube-proxy
Quản lý các luật mạng (IPtables hoặc IPVS) để mở luồng kết nối cho các dịch vụ Service.
containerd
Container Runtime chịu trách nhiệm kéo image từ Docker Registry và quản lý việc chạy container.

Block 01 · Chu Kỳ Tài Nguyên

Quy Trình Khởi Tạo & Chạy Pod.

Từ lúc gọi lệnh apply cho đến khi container thực tế hoạt động trên hạ tầng.

1
Gửi khai báo qua kubectl apply
File YAML được gửi đến kube-apiserver. API-server kiểm tra định dạng, phân quyền RBAC và ghi trạng thái này vào cơ sở dữ liệu etcd.
2
Tạo tài nguyên logic bằng Controller
Deployment Controller phát hiện tài nguyên mới, ra lệnh tạo ra ReplicaSet, sau đó ReplicaSet sinh ra các Pod có trạng thái Pending.
3
Lập lịch tìm Node từ Scheduler
kube-scheduler quét các Pod chưa có Node chỉ định. Chọn Node tốt nhất dựa trên RAM/CPU trống, ghi nhận Node được gán vào spec của Pod.
4
Thực thi tại Node qua Kubelet
kubelet tại Node đích phát hiện Pod gán cho mình, ra lệnh cho containerd pull image và khởi chạy container. Pod chuyển trạng thái Running.

Block 01 · Đối Tượng Cơ Bản

Pod & Các Mẫu Container Phối Hợp.

Pod là đơn vị tính toán nhỏ nhất. Nhiều container trong cùng Pod chia sẻ chung IP, localhost, IPC và Volume.

Mẫu Container Phụ Trợ (Patterns)
Sidecar Pattern
Chạy container phụ cạnh app chính trong cùng Pod để hỗ trợ (ví dụ: Fluent Bit đọc log từ file ghi ra chung rồi đẩy lên Elasticsearch).
Init Container
Chạy và kết thúc thành công trước khi các container chính được khởi tạo (ví dụ: chạy script chờ DB sẵn sàng).
# YAML đặc tả Pod có Init Container apiVersion: v1 kind: Pod metadata: name: web-app spec: initContainers: - name: wait-db image: busybox command: ["sh", "-c", "until nc -z db 5432; do sleep 2; done"] containers: - name: nginx image: nginx:1.27

Block 01 · Điều Phối

Deployment & ReplicaSet.

Deployment quản trị vòng đời Pod thông qua việc quản lý các bản ghi ReplicaSet tương ứng.

Cấu Trúc Quản Lý Phân Cấp
Deployment quản lý các revision của ReplicaSet. ReplicaSet trực tiếp duy trì số lượng Pod.
Cơ chế Tự Phục Hồi (Self-Healing)

Duy trì Replicas liên tục: Nếu một Pod bị xóa đột ngột hoặc Node sập, ReplicaSet Controller phát hiện số lượng Pod thực tế nhỏ hơn mong muốn (2 < 3), lập tức tạo mới 1 Pod thay thế để khôi phục trạng thái khai báo ban đầu.

Insight Vận Hành
Deployment giúp rời bỏ cách quản lý Pod đơn lẻ, cung cấp năng lực kiểm soát version và rollback không downtime.

Namespace & Labels.

Vách ngăn phân chia logic và cơ chế định danh gắn kết linh hoạt các tài nguyên trong cụm.

Namespace (Vách ngăn logic)
Phân tách tài nguyên trong cùng cụm vật lý. Phù hợp cách ly môi trường (dev, staging, prod) hoặc đội nhóm.

Cảnh báo thực tế: Namespace không cách ly traffic mạng. Các Pod khác Namespace vẫn gọi chéo được nhau trừ khi áp dụng NetworkPolicy.

Labels & Selectors (Định danh)
Cặp Key-Value gán trực tiếp lên metadata của tài nguyên. Selectors được sử dụng để truy vấn hoặc kết nối các tài nguyên với nhau (như Service tìm Pods).
# Lọc Pod theo label kubectl get pods -l app=frontend,env=prod

Block 02 · Cấu Hình

ConfigMap vs Secret.

Tách biệt mã nguồn ứng dụng ra khỏi dữ liệu cấu hình để đảm bảo nguyên lý Cloud-native.

Định Nghĩa Tài Nguyên
ConfigMap
Lưu trữ dữ liệu cấu hình không bảo mật dưới dạng key-value hoặc file cấu hình raw (config.json, env vars).
Secret
Lưu trữ thông tin nhạy cảm (mật khẩu, tokens, SSH keys) đã được mã hóa dạng base64 trong manifest.
Cảnh báo Bảo mật Thực chiến
Dữ liệu trong Kubernetes Secret mặc định chỉ encode Base64, không phải mã hóa. Bất kỳ ai có quyền đọc API hoặc file YAML đều decode được thông tin dễ dàng.

Giải pháp Production: Kích hoạt Encryption-at-rest cho etcd hoặc sử dụng external secret manager (Vault, AWS Secrets Manager thông qua ESO).

Headless Service (clusterIP: None)

Ingress: Định Tuyến Lớp Ứng Dụng (L7).

Ingress quản lý định tuyến HTTP/HTTPS từ ngoài internet vào các Service bên trong cụm dựa trên Host hoặc Path.

Ưu điểm so với L4 LoadBalancer
Tiết kiệm chi phí
Chỉ cần khởi tạo duy nhất 1 Load Balancer của Cloud làm Gateway, định tuyến nhiều service qua Path.
Tính năng L7

Block 03 · Độ Tin Cậy

Health Probes: Cơ Chế Giám Sát Sức Khỏe.

Kubernetes giám sát và quyết định trạng thái hoạt động của Container thông qua 3 loại probe.

1. Startup Probe
Kiểm tra ứng dụng khởi chạy thành công hay chưa. Tạm dừng các probe khác (Liveness, Readiness) cho đến khi startup thành công.
2. Liveness Probe
Kiểm tra container còn sống không. Thất bại liên tiếp sẽ kích hoạt cơ chế giết container hiện tại và khởi tạo container mới.
3. Readiness Probe
Kiểm tra ứng dụng đã sẵn sàng nhận traffic chưa. Thất bại sẽ gỡ Pod khỏi danh sách endpoints của Service (không restart).

Bẫy thiết kế hệ thống: Phải tách biệt endpoint Liveness và Readiness. Nếu database sập, Readiness trả về fail để dừng nhận traffic vào app, nhưng Liveness phải trả về success để tránh restart app liên tục vô ích.

Block 03 · Tài Nguyên

Quản Lý Tài Nguyên & Tự Động Scale.

Giới hạn CPU/RAM giúp bảo vệ Worker Node khỏi tình trạng cạn kiệt tài nguyên cục bộ.

Requests vs Limits (RAM/CPU)
Requests
Lượng tài nguyên tối thiểu cần để khởi chạy Pod. Scheduler dựa vào đây để gán Node.
Limits
Ngưỡng tối đa Pod được dùng. Vượt CPU gây throttled (chạy chậm). Vượt Memory kích hoạt OOM và gây OOMKilled.
HPA (Horizontal Pod Autoscaler)
Tự động tăng/giảm số lượng bản sao Pod dựa trên mức sử dụng CPU/RAM thực tế hoặc metric tùy biến của ứng dụng.
Cần cấu hình chính xác resource requests vì HPA dựa trên tỷ lệ phần trăm sử dụng của thông số requests để ra quyết định scale.

Rollout & Rollback: Cập Nhật Zero-Downtime.

Chiến lược Rolling Update mặc định thay thế dần từng Pod cũ bằng Pod mới một cách tuần tự.

Quy Trình Rolling Update
Bước 1
Trạng thái V1
Hệ thống đang chạy ổn định với 3 Pods phiên bản cũ (V1).
Bước 2
Khởi động Pod V2
Khởi động 1 Pod V2 mới. Chỉ gỡ Pod V1 sau khi Pod V2 đạt trạng thái Ready.
Bước 3
Hoàn tất
Lặp lại tuần tự cho đến khi toàn bộ cụm chuyển sang V2.
Khôi phục Sự Cố (Rollback)
Lịch sử cập nhật được lưu giữ để có thể hoàn tác tức thì.

StatefulSet vs DaemonSet.

Các loại tài nguyên chuyên dụng cho bài toán lưu trữ trạng thái ổn định và công cụ giám sát hạ tầng.

StatefulSet (Ứng dụng có trạng thái)
Chạy các hệ thống quản trị dữ liệu (PostgreSQL, Redis Cluster, Kafka).
  • Gán tên định danh số thứ tự duy nhất cố định: db-0, db-1.
  • Mỗi Pod liên kết tĩnh với PVC riêng, giữ nguyên liên kết kể cả khi Pod bị xóa và dựng lại.
DaemonSet (Chạy trên mọi Node)
Chạy các agent giám sát hoặc thu thập log hệ thống (Fluent Bit, Prometheus Node Exporter).
  • Đảm bảo chạy duy nhất 1 bản sao Pod trên tất cả các Worker Node.
  • Tự động chạy Pod trên Node mới thêm vào cụm khi scale-out hạ tầng.
Block 03 · Operations & Storage Slide 19
Block 03 · Lưu Trữ

PV, PVC & StorageClass.

Kiến trúc tách biệt vai trò quản lý hạ tầng phần cứng của Admin và yêu cầu sử dụng của Developer.

PersistentVolume (PV)
Tài nguyên lưu trữ thực tế được cấp phát trong hệ thống vật lý hoặc cloud (AWS EBS, EFS, NFS volume).
PersistentVolumeClaim (PVC)
Yêu cầu xin cấp phát tài nguyên lưu trữ từ phía lập trình viên (xin dung lượng, chế độ truy cập).
StorageClass (SC)
Định nghĩa driver tự động khởi tạo PV vật lý khi có yêu cầu PVC gửi lên (Dynamic Provisioning).

Chế Độ Truy Cập Lưu Trữ & Hạn Chế.

Phân loại khả năng kết nối đồng thời từ các Node tới tài nguyên lưu trữ.

Access Mode Viết tắt Mô tả khả năng kết nối Hỗ trợ thực tế Hạn chế (Drawbacks)
ReadWriteOnce RWO Chỉ cho phép 1 Node kết nối để đọc và ghi dữ liệu AWS EBS, Disk nội bộ Node Không chia sẻ được file cho các Pod nằm trên các Node khác nhau.
ReadOnlyMany ROX Nhiều Node kết nối song song nhưng chỉ được phép đọc AWS EFS, NFS Các Pod không thể ghi dữ liệu mới trực tiếp lên volume này.
ReadWriteMany RWX Nhiều Node kết nối đồng thời để cùng đọc và ghi AWS EFS, NFS, CephFS Cần cấu hình khoá file phức tạp ở mức app để tránh ghi đè hư hỏng.

Block 04 · Bảo Mật

RBAC & NetworkPolicy.

Bảo vệ cả đường API điều phối lẫn đường truyền mạng nội bộ giữa các container trong cụm.

RBAC (Role-Based Access Control)
ServiceAccount
Định danh gán cho các Pod để ứng dụng giao tiếp hợp lệ với API-server.
Role
Khai báo quyền hạn thao tác (verbs: get, list, create) trên tài nguyên trong namespace.
RoleBinding
Bản ghi liên kết tài khoản ServiceAccount với Role để thực thi phân quyền.
NetworkPolicy (Tường lửa Pod)
Quản lý chặn/cho phép các luồng truyền mạng giữa các Pod dựa trên Labels và IP blocks.

Bảo mật mặc định: Kubernetes mặc định cho phép mọi Pod kết nối chéo tự do. Cấu hình NetworkPolicy là bắt buộc để xây dựng mạng Zero-Trust.
