Sprint 2 · Day 2

Scenario C -
Unclear Leader

Leadership failure, team misalignment, and emergency re-alignment after task assignment moved from Jira into Slack.

No Jira Ticket = No Task For PMs, Scrum Masters, Team Leads, Agile Coaches, and Delivery Teams
Case Frame

What happened in Sprint 2?

The problem is not effort or intent. The failure is that work entered the sprint without a governed source of truth.

09:10
Leader posts in Slack
Assignments are shared as chat messages, not Jira tickets with acceptance criteria and ownership.
10:25
Dev1 starts wrong path
A vague request is interpreted differently from what the leader intended.
11:40
Dev2 refactors
Refactoring begins without official assignment, priority check, or sprint impact review.
13:30
QA writes early tests
QA prepares tests for a feature that has not been built and may not match the final scope.
Key diagnosis: nobody is obviously wrong. The system allowed reasonable people to make incompatible decisions.
Block 01 · SituationSlide 02
Leadership Failure

The root cause is unclear task governance.

Slack is useful for discussion, clarification, and escalation. Jira is the operating record for committed sprint work.

1
Source of truth
The team needs one place where task scope, owner, priority, status, and acceptance criteria are visible.
0
Intentional faults
The PM should avoid blaming individuals and focus on closing the process gap immediately.
3
Misaligned streams
Dev1, Dev2, and QA are all productive locally but disconnected from the sprint plan globally.

No Jira Ticket = No Task

Chat can trigger discussion, but work starts only after the ticket exists, has an owner, and has enough definition to verify done.

Slack != Assignment

Block 02 · Role ImpactSlide 05
Decision Record

What was missing from Jira?

A good ticket reduces interpretation variance. It makes scope reviewable before the team spends effort.

Missing ticket fields
Owner
Who is accountable for doing the work now.
Outcome
What user or system behavior must change.
Acceptance
Observable conditions for Done and QA validation.
Priority
Why this task belongs in Sprint 2 ahead of other work.
Ambiguity test
  • Can two developers read the ticket and build the same outcome?
  • Can QA derive tests without guessing feature behavior?
  • Can the Scrum Master see whether the task is sprint work or backlog work?
  • Can the Product Owner confirm whether the task supports sprint goals?

Immediate Response

The first 30 minutes are procedural, not emotional.

Move quickly, keep the tone neutral, and make every decision visible to the whole team.

Block 03 · Team TriageSlide 09
Jira Governance

Define the working agreement as executable rules.

The team should not rely on memory or personality. The workflow should make the right behavior obvious.

Team rulebook
  • No Jira ticket = no task.
  • Vague task = sender clarifies, receiver asks.
  • Slack decisions must be summarized into Jira within the same working session.
  • Unassigned work is backlog work until the board says otherwise.
Suggested Jira workflow
Backlog -> Ready owner + acceptance criteria -> In Progress assigned sprint work only -> Ready for QA feature built and reviewable -> Done tests passed + evidence linked

The status gates prevent QA from testing imaginary behavior and prevent developers from silently expanding scope.

Block 03 · Jira RulesSlide 10
Prevention

Prevent recurrence with lightweight controls.

The fix is not more meetings. It is tighter entry criteria, visible ownership, and faster clarification loops.

Control Owner Frequency Success signal
Definition of Ready PM / Scrum Master Before sprint commitment Ticket has owner, value, scope, acceptance criteria, and priority.
Daily board walk Scrum Master Daily standup Every person speaks from Jira status, not from private chat memory.
Slack-to-Jira capture Message sender Same working session Important decisions are linked to the relevant ticket.
Assignment audit Team Lead Twice per sprint No active work exists without an assigned Jira issue.
