01 · Kiểm soát

Branch Protection & Secrets Management.

Ngăn chặn cấu hình sai sót và loại bỏ hoàn toàn các thông tin đăng nhập tĩnh khỏi mã nguồn.

Branch Protection Rules
  • Require status checks: Buộc CI workflow (test/lint) phải pass mới cho phép merge.
  • Require approval: Ngăn tự ý merge bằng cách yêu cầu tối thiểu N đồng nghiệp duyệt.
  • Restrict force pushes: Khóa quyền ghi đè history lên các nhánh chính.
Secrets Management
  • Environment Secrets: Phân tách secret của staging và production riêng biệt.
  • OIDC Provider: Trao đổi token ngắn hạn qua OIDC với AWS/Azure IAM Role.
  • Keyless Authentication: Triệt tiêu rủi ro lộ credential tĩnh khi CI bị thỏa hiệp.
Khuyến nghị thực tế
Tuyệt đối không commit credentials dạng plaintext lên Git. Sử dụng GitHub OIDC liên kết động với Cloud IAM Role thay vì dùng static Access Key.

Triết lý hạ tầng GitOps Fundamentals.

Git là nguồn sự thật duy nhất, cơ chế đồng bộ Pull-based và vòng lặp hòa giải.

02 · Nguyên tắc

Git là Nguồn Sự Thật Duy Nhất.

Trạng thái mong muốn của toàn bộ hệ thống được khai báo tường minh dưới dạng code lưu trong Git.

✓ Tuân thủ GitOps
  • Cập nhật tag Docker image bằng commit/PR lên Git.
  • Thay đổi replica count bằng cách sửa manifest file trong Git repo.
  • Thực hiện Rollback hệ thống bằng lệnh git revert.
  • Lịch sử vận hành (Audit log) nằm hoàn bộ trên lịch sử commit.
✗ Vi phạm GitOps
  • Sử dụng lệnh kubectl apply hoặc kubectl edit trên cluster.
  • Sửa thông số qua Kubernetes Dashboard hoặc K9s trực tiếp.
  • CI pipeline tự động deploy thẳng lên cluster không cập nhật Git.
  • Mất dấu vết lịch sử thay đổi tài nguyên thực tế trên cluster.
Quy tắc bất biến
Tuyệt đối không thao tác ad-hoc trực tiếp trên cluster. Mọi thay đổi trạng thái hệ thống bắt buộc phải bắt đầu bằng một Git commit.

02 · Kiến trúc

Mô hình Push-based vs Pull-based Deployment.

So sánh kiến trúc vận hành giữa pipeline đẩy trực tiếp vào cluster và GitOps controller kéo desired state từ Git.

Đặc điểm so sánh Push-based (Truyền thống) Pull-based (GitOps) Hạn chế chính (Drawback)
Nơi thực thi Từ bên ngoài (CI Server như GitHub Actions) Từ bên trong (Agent chạy trực tiếp trong Cluster) Tiêu hao tài nguyên chạy Agent trong Cluster.
Quản lý RBAC CI Server thường cần kubeconfig hoặc token có quyền ghi vào Cluster API Agent cần quyền đọc Git và RBAC ghi đúng phạm vi trong cluster Vẫn phải giới hạn quyền agent bằng namespace, AppProject và policy phù hợp.
An toàn cổng API Cần đường mạng từ CI runner tới Kubernetes API server CI không cần gọi Kubernetes API; agent trong cluster gọi ra Git và gọi API nội bộ Khi khẩn cấp vẫn có thể dùng kubectl, nhưng phải cập nhật Git để tránh drift.
Tự động khôi phục Không phát hiện lệch cấu hình (Drift) Hòa giải tự động liên tục khi phát hiện drift Trễ đồng bộ phụ thuộc chu kỳ quét của Agent.

02 · Cơ chế đồng bộ

Vòng lặp Hòa giải & Drift Detection.

Cơ chế phát hiện và xử lý sai lệch cấu hình tự động của GitOps Agent.

Cơ chế Self-Healing (Tự phục hồi)
  • Drift Detection: Quét định kỳ so sánh Actual State trong Cluster với Desired State trong Git.
  • Self-Healing: Nếu phát hiện thay đổi thủ công ngoài luồng, Agent tự động ghi đè về đúng trạng thái trên Git.
  • Chu kỳ đồng bộ: Mặc định đồng bộ tự động mỗi 3 phút hoặc đồng bộ tức thì thông qua Webhook.

Argo CD vs Flux v2: So sánh & Lựa chọn.

Phân tích chuyên sâu kiến trúc monolithic trực quan vs Unix-style mỏng nhẹ.

03 · Phân tích

Hai triết lý thiết kế GitOps Agents.

So sánh triết lý quản trị tập trung trực quan và thiết kế Kubernetes-native tối giản.

Argo CD — Tập trung & Trực quan
  • Cung cấp Web UI trực quan cho resource tree, sync status, health và thao tác vận hành.
  • Phù hợp cho mô hình multi-tenancy nhờ AppProject và Argo CD RBAC.
  • Giúp Developers tự theo dõi trạng thái ứng dụng mà không cần bắt đầu từ kubectl.
Flux v2 — Mỏng nhẹ & Micro-controllers
  • Thiết kế Unix-style, chia nhỏ thành các controller độc lập đơn nhiệm (source, helm).
  • Quản lý CLI-first, tích hợp sâu vào kiến trúc Kubernetes Native và CRD của cluster.
  • Rất thích hợp cho hạ tầng Edge Computing hoặc IoT cần tối ưu tối đa dung lượng RAM/CPU.
Điểm cốt lõi cần nhớ
Argo CD mạnh về trải nghiệm quan sát và thao tác qua UI; Flux v2 mạnh ở mô hình Git-first, Kubernetes-native và automation dạng controller nhỏ.

03 · Đánh giá

Ma trận So sánh kỹ thuật & Quyết định.

Lựa chọn giải pháp GitOps Agent tối ưu theo nhu cầu vận hành thực tế của dự án.

Tính năng Argo CD Flux v2 Hạn chế chính (Drawback)
Web UI Web UI mạnh, hiển thị live status, resource tree và thao tác sync. Không có UI chính thức (dùng extension ngoài). Web UI của Argo CD làm tăng bề mặt vận hành cần bảo vệ và tiêu thụ thêm tài nguyên.
Multi-tenancy Tập trung qua AppProject và Argo CD RBAC. Phân tách an toàn bằng Namespace RBAC mặc định. Cả hai đều cần thiết kế RBAC cẩn thận; Flux thiên về Kubernetes-native RBAC.
Helm Charts Render chart thành yaml tĩnh tại controller rồi apply. Dùng Helm Controller thực hiện đúng quy trình Helm. Argo CD render Helm chart như manifest; không tạo Helm release metadata để quản lý bằng helm list.
Auto Image Update Cần cài thêm addon argocd-image-updater. Tích hợp sẵn qua image-automation-controller. Addon phụ trợ có độ trễ lớn khi quét và đẩy tag mới.
Quy tắc quyết định chọn công cụ
Chọn Argo CD khi cần UI để demo, trace resource tree và xem sync/health nhanh. Chọn Flux v2 khi muốn mô hình Git-first nhẹ hoặc cần Image Automation Controller cập nhật tag mới vào Git.

Điều phối ứng dụng Argo CD Advanced.

04 · Quản lý khai báo

Mẫu thiết kế App-of-Apps Pattern.

Sử dụng root Application để Argo CD quản lý vòng đời của nhiều child Application trong cùng repo.

Cấu trúc App-of-Apps trong repo W9
cloud/w9/day-a/argocd/
├── app-of-apps.yaml          # Root app: w9-root
├── w8-platform-app.yaml      # Child app wave 0
├── w9-observability-app.yaml # Child app wave 1
└── w9-rollout-app.yaml       # Child app wave 2

cloud/w9/lab/
├── platform/                 # Namespace, config, secret, service
├── observability/            # OTel config, PrometheusRule
└── rollout/                  # Rollout, AnalysisTemplate
Chỉ cần apply app-of-apps.yaml. Root app đọc ba child app YAML trong thư mục argocd; thêm/xóa child app bằng commit Git.
Root Application Manifest trong W9
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: w9-root
  namespace: argocd
spec:
  source:
    repoURL: https://github.com/MinhKhoa2209/minhkhoa-aws-accelerator-p2.git
    targetRevision: HEAD
    path: cloud/w9/day-a/argocd
    directory:
      recurse: false
      include: "{w8-platform-app.yaml,w9-observability-app.yaml,w9-rollout-app.yaml}"
Lưu ý cấu hình bắt buộc
Thiết lập prune: true để khi YAML của child app bị xóa khỏi Git, Argo CD có thể thu hồi tài nguyên tương ứng trong lần sync kế tiếp.

04 · Tác vụ đặc biệt

Sync Hooks: Can thiệp chu trình đồng bộ.

Thực hiện các tác vụ đặc biệt (DB Migration, Notify) tại các mốc thời gian của chu trình đồng bộ.

Các loại Sync Hooks phổ biến
Hook Thời điểm Mục đích Hạn chế
PreSync Trước apply wave Chạy DB Migration Job lỗi sẽ chặn đứng deployment.
PostSync Sau wave healthy Smoke tests, Slack notify Không ngăn được lỗi phát sinh trong sync.
SyncFail Sync thất bại Dọn dẹp rác, cảnh báo Khó thiết kế kịch bản dọn dẹp hoàn hảo.
Ví dụ Database Migration (PreSync)
apiVersion: batch/v1
kind: Job
metadata:
  name: db-migration-job
  annotations:
    argocd.argoproj.io/hook: PreSync
    argocd.argoproj.io/hook-delete-policy: HookSucceeded
spec:
  template:
    spec:
      restartPolicy: Never
Mẹo dọn dẹp tài nguyên
Sử dụng annotation hook-delete-policy: HookSucceeded để tự động xóa Pod của Job sau khi chạy thành công, tránh lãng phí tài nguyên K8s.

Chiến lược khôi phục Rollback Strategy.

Xử lý sự cố production an toàn và ngăn ngừa rủi ro ghi đè từ cơ chế tự động của Agent.

05 · Khôi phục sự cố

Bản đồ 4 cấp độ Khôi phục hệ thống.

Khi xảy ra sự cố Production, kỹ sư có 4 phương án khôi phục trạng thái với các đánh đổi khác nhau.

Git Level
Git Revert
Dùng git revert tạo commit phủ định lỗi và push lên main. Đây là rollback chuẩn vì Git và cluster vẫn cùng desired state.
K8s Level
Rollout Undo
Chạy kubectl rollout undo khôi phục nhanh ReplicaSet. Hữu ích khi khẩn cấp, nhưng tạo drift nếu Git chưa được sửa.
Helm Level
Helm Rollback
Dùng helm rollback đưa release về phiên bản cũ. Nếu release do GitOps quản lý, cần commit trạng thái rollback vào Git.
Agent Level
Argo CD UI
Bấm Rollback trên Argo CD UI để đổi live state theo revision cũ. Vẫn cần cập nhật Git để tránh drift lâu dài.
Cảnh báo Drift nghiêm trọng
Nếu Git vẫn giữ version lỗi và Argo CD đang auto-sync/self-heal, thao tác ad-hoc như kubectl rollout undo hoặc helm rollback có thể bị reconcile quay lại version trong Git. Cứu runtime xong phải sửa Git ngay.

05 · Đánh giá chiến lược

So sánh kỹ thuật các cơ chế Rollback.

Lựa chọn giải pháp khôi phục phù hợp dựa trên mức độ khẩn cấp và tính toàn vẹn của Git.

Phương pháp Tốc độ khôi phục Toàn vẹn Git Khuyên dùng Rủi ro chính (Drawback)
Git Revert Trung bình (chờ PR/merge và Argo CD sync) ✅ Đạt 100% (có audit log) ✅ Khuyến nghị hàng đầu Chậm hơn thao tác runtime, nhưng có audit log và không tạo drift.
kubectl rollout undo Cực nhanh (vài giây) ❌ Không (Gây drift) ⚠️ Chỉ dùng khẩn cấp Nếu Git vẫn giữ version lỗi, Argo CD có thể sync cluster quay lại version lỗi.
Helm Rollback Nhanh (dưới 1 phút) ❌ Không (Gây drift) ⚠️ Chỉ dùng khẩn cấp Gây drift nếu Helm release đang được quản lý bởi GitOps; cần commit rollback vào Git.
Argo CD UI Rollback Nhanh (dưới 1 phút) ❌ Lệch tạm thời ⚠️ Chỉ dùng khẩn cấp Không thay thế được rollback bằng Git; cần commit/revert để desired state phản ánh trạng thái mong muốn.
Cảnh báo Vận hành
Có thể can thiệp trực tiếp để cứu sự cố runtime, nhưng phải cập nhật Git ngay sau đó. Nếu desired state trong Git vẫn là bản lỗi, Argo CD có thể reconcile cluster quay lại bản lỗi.
